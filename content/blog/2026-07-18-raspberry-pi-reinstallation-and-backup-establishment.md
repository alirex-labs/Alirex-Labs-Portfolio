## はじめに：インフラ喪失から完全復旧へのドラマ

Alirex Labsの基盤インフラである社内サーバー `alirex-labs` (Raspberry Pi 5) が、OSの再インストールに伴い全データが初期化されるという重大な危機に直面しました。
バックアップが存在しない「クリーンな状態」からの復旧という過酷な前提のもと、全15サービス（5スタック）の再構築から、接続エラーの解決、さらには将来を見据えた自動バックアップ体制の確立（PRJ-0017）まで、本日のDMO AgentとCOO Agentの連携によるインフラ復旧劇をお届けします。

---

## 1. 15コンテナサービスの一斉復旧とGitea連携の苦闘

Docker Compose統合管理ツール「Dockge」を用いて分割された5つのスタックを順番に起動していきましたが、いくつかの壁が立ちはだかりました。

### ① SSHホスト鍵の不一致（警告発生）
再構築されたラズパイに対して、ローカルPCからGitプッシュを試みた際、`WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!` という深刻な警告が発生しました。
*   **原因**: ラズパイの物理的な再構築によりSSHサーバーのホスト鍵が変わってしまったため、ローカルPC側の `known_hosts` に残っていた旧鍵との食い違いが生じ、接続がブロックされました。
*   **解決**: ローカルPCのPowerShellから `ssh-keygen -R "[alirex-labs]:2222"` を実行してキャッシュをクリア。新しい鍵を受け入れることで開通しました。その後、認証エラーを素早く解決するため、ローカルのリモートURLを一時的にHTTP接続に切り替え、無事に開発用リポジトリの全履歴とタグをGitea上へ復元しました。

### ② Gitea Webhookのローカルブロック問題（SSRF対策）
リポジトリのプッシュを検知してドキュメントポータル（MkDocs）を同期する `git-sync` Webhookが、接続エラーで動作しませんでした。
*   **原因**: Gitea v1.16以降のデフォルトセキュリティ仕様により、Dockerコンテナ間やLAN内のプライベートIPへのWebhook送信がSSRF攻撃対策としてブロックされていました。
*   **解決**: ラズパイ上のGitea設定ファイル（`app.ini`）に `[webhook] ALLOWED_HOST_LIST = *` を追記してコンテナを再起動。これによりコンテナ間のWebhook疎通が確認され、緑色のチェックマークが表示されるようになりました。

## 2. Web Previewの復旧とポータルサイトの「再進化」

Web Preview（ポート8080）にアクセスした際、空ディレクトリがマウントされていたために `403 Forbidden` となる問題が起きていました。
ローカルPCのワークスペースに保管されていたHTMLアセットをラズパイにSCP転送して復旧を行いましたが、ただ元の状態に戻すだけでなく、DMO Agent主導のもとでUI/UXの **「再進化」** を実施しました。

*   **マウス追従発光エフェクト (Radial Glow)**: マウスカーソルの動きに合わせて、カードがガラスのように綺麗に発光する効果を適用。
*   **Staggeredアニメーション**: ページロード時に、各カードが少しずつ時間差（ディレイ）を持って下から浮き上がるフェードイン演出を導入。
*   **最新状況の同期**: プロジェクト台帳に新しく登録された `PRJ-0016` (再インストール) と `PRJ-0017` (バックアップ) の情報をポータルへ反映。

## 3. 「転んでもただでは起きない」バックアッププロジェクトの起票

今回の「バックアップ不在による手動復旧のオーバーヘッド」を教訓とし、Alirex Labsはただ復旧するだけで終わらせません。将来的な障害に対して完全に自律復旧できる仕組みを作るため、新規プロジェクト **`PRJ-0017` (Infrastructure-Backup-Establishment)** を正式に起票しました。

### プロジェクトの目的と概要
*   PostgreSQLデータベース（5つのDB）の自動ダンプスクリプト (`backup_db.sh`) の設計と日次実行 (Cron)。
*   Giteaリポジトリ、Passbolt暗号鍵（GPG/JWT）、Paperless-ngx PDFなどの重要データの自動アーカイブ管理。
*   ラズパイローカルへの保存（過去7日分の世代管理）に加え、外部（ローカルPCやクラウド、またはNAS等）への自動転送体制の確立。
*   有事の際に一瞬で元に戻すための **「リストア（復旧）手順書」** の定義と、別環境への手動復旧検証の実施。

本プロジェクトの起票に伴い、プロジェクト憲章（`Project-Charter.md`）を作成し、公式のプロジェクト台帳（`Project-Registry.md`）を `Version 2.55` へバンプして正式に開発を開始しました。

## おわりに：知識を資産化する「企業OS」の強み

今回のインフラ障害は一時的な痛みを伴いましたが、そのトラブルシューティングや解決策はすべてナレッジベース（KB）に蓄積され、有事に備えた自動バックアップ体制という強固な仕組みへと昇華されました。
トラブルを機会と捉え、再利用可能な資産へと変えていく。これこそが、Alirex AI Labsが提唱する「1人 × AI組織 = 100人規模の企業力」を支える企業OSの進化スピードです。
